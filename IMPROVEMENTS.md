## 1. 페이지 접속할 때마다 재입력 → 정보 저장(앱처럼)

### 현상
접속할 때마다 닉네임/그룹 온보딩을 다시 거쳐야 한다.

### 원인 진단 (⚠️ 규명 먼저)
코드상으로는 **이미 프로필이 저장된다.**
- `src/lib/profile.ts` — 온보딩 성공 시 `localStorage['wt_profile']`에 프로필 캐시.
- `src/lib/supabase.ts` — 익명 세션을 `persistSession: true`로 localStorage에 유지.
- `src/context/ProfileContext.tsx` — 부팅 시 `loadCachedProfile()`로 캐시를 읽어 게이트를 통과.

따라서 **같은 브라우저라면 재온보딩이 나오지 않아야 정상**이다. 매번 나온다면 다음 중 하나가 실제 원인일 가능성이 높다 — **재현 조건을 먼저 확인해야 한다.**

가능성 (높은 순):
1. **iOS Safari ITP** — 홈화면에 추가하지 않은 사이트의 localStorage를 "7일 미접속 시" 자동 삭제. PWA 미설치 상태(Phase 6 미완)라 특히 취약.
2. **홈화면 저장소 격리** — Safari 탭에서 온보딩 → 홈화면 아이콘으로 열면 별도 저장소 컨텍스트라 캐시가 비어 있음.
3. **사파리 프라이빗 모드 / 시크릿 탭** — localStorage가 세션 종료 시 소멸.
4. **익명 세션 만료** — 캐시(`wt_profile`)는 남아도 익명 JWT가 만료되어 RPC/조회가 401 → 사실상 재시작. (단 현재 게이트는 캐시만 보고 통과시키므로 이 경우 앱 내부에서 데이터 로드 실패로 나타남)

> **먼저 할 일**: 실제 배포 URL을 어떤 환경(iOS/Android/PC, 사파리 탭/홈화면 아이콘)에서 열었는지 확인하고, 재현되는 기기에서 DevTools/원격 디버깅으로 `localStorage['wt_profile']`와 `supabase.auth.token`이 유지되는지 관찰한다.

### 개선안
"진짜 앱처럼" 유지되려면 **저장소 지속성 강화 + PWA 설치 + 복구 무마찰**을 함께 간다.

1. **PWA 설치 유도 (근본 해결, Phase 6와 통합)**
   - `manifest.json` + 아이콘 + 서비스워커 추가 → 홈화면에 설치되면 ITP 저장소 삭제 대상에서 제외되고 저장소 컨텍스트가 통일된다.
   - iOS는 홈화면 추가 안내 배너(첫 방문 후), Android/데스크톱은 `beforeinstallprompt` 캡처해 "앱 설치" 버튼 노출.
2. **저장소 이중화** — 프로필 캐시를 localStorage 단독이 아니라, 가능하면 IndexedDB에도 미러링(더 오래 살아남음). 최소한 부팅 시 캐시가 비어 있고 익명 세션은 살아 있으면 서버에서 내 프로필을 재조회해 캐시를 복구.
3. **세션 유효성 자가복구** — 부팅 시 `getSession()` 후 만료됐으면 조용히 `refreshSession()` 시도, 실패 시에만 온보딩. 캐시가 있는데 세션만 죽은 경우 "복구 코드로 복구" 화면을 우선 노출(재온보딩 강제 X).
4. **복구 무마찰 백업** — 온보딩 완료 시 발급되는 복구 코드를 사용자가 잃지 않도록: (a) 최초 발급 화면에서 "복사/캡처" 강한 유도, (b) 설정 화면에 항상 복구 코드 재확인 진입점. (현재 발급 시 1회 노출 정책은 유지)

### 작업 항목
- [ ] 배포 환경에서 재현 조건 확정 (iOS 홈화면 여부, ITP 여부)
- [ ] `manifest.json` + 아이콘 세트 + 서비스워커(앱셸 캐싱) 추가 → 설치형 PWA
- [ ] 설치 유도 UI (iOS 안내 배너 / `beforeinstallprompt`)
- [ ] 부팅 시 "캐시 없음 + 익명세션 살아있음 → 서버에서 프로필 재조회" 복구 경로
- [ ] 설정 화면: 내 그룹 초대코드 / 복구 코드 재확인 진입점

## 2. 운동 시작 / 운동 완료 버튼 + 경과 타이머 자동 시작

### 현상
운동 세션에 명시적 "시작/완료" 개념이 없다. 세션은 첫 운동을 추가할 때 조용히 생성되고(`LogPage.handlePick → createSession`), 총 운동 소요시간을 측정하지 않는다.

### 원인 진단
- `WorkoutSession` 타입/테이블에 `started_at`, `ended_at`이 없음 (`date`만 존재 — `src/lib/types.ts:35`).
- `HomePage`의 "운동 시작" 버튼은 단순히 `/log`로 이동만 한다(세션 생성/타이머 없음).

### 개선안
1. **스키마 확장** — `workout_sessions`에 `started_at timestamptz`, `ended_at timestamptz` 추가.
2. **명시적 시작** — "운동 시작"을 누르면 세션 생성 + `started_at = now()`. (첫 운동 추가로 지연 생성되던 흐름은 시작 버튼으로 대체하거나, 시작 안 하고 운동 추가 시 그 시점을 시작으로 간주)
3. **경과 타이머** — 상단 고정 바에 `now - started_at` 경과시간 표시. **`RestTimer`와 동일하게 `Date.now()` 절대시각으로 계산** → 탭 이탈/백그라운드/새로고침에도 정확 (경과값을 상태로 들지 않고 `started_at`만 저장).
4. **명시적 완료** — "운동 완료"를 누르면 `ended_at = now()` 저장, 타이머 정지, 요약 화면(총 시간·볼륨·세트 수·PR) 노출. 완료 후 그대로 "피드에 공유" 유도.
5. **미완료 세션 이어하기** — `started_at` 있고 `ended_at` 없으면 홈에서 "운동 중 (23:14 경과) — 이어서" 상태로 표시.

> UX 참고: `RestTimer`(휴식, 카운트다운)와 **운동 경과 타이머(카운트업)는 별개**다. 둘을 동시에 상단에 겹치지 않게 레이아웃 정리 필요(경과는 상단, 휴식은 하단 등).

### 작업 항목
- [ ] 마이그레이션: `workout_sessions.started_at`, `ended_at` 추가
- [ ] `createSession`에 `started_at` 세팅, `endSession(id)` 추가 (`src/lib/workouts.ts`)
- [ ] `WorkoutSession` 타입에 필드 추가
- [ ] `ElapsedTimer` 컴포넌트 (카운트업, `Date.now()` 기반)
- [ ] `LogPage` 상단에 시작/완료 버튼 + 경과 타이머, 완료 시 요약 모달
- [ ] `HomePage`: 미완료 세션 "이어서" 상태 반영
- [ ] (선택) 완료 소요시간을 피드 카드/기록 열람에 표시

## 3. 캘린더 + 날짜별 기록 상세 (레퍼런스 반영)

> 첨부 레퍼런스(fleek 스타일) 3장 기준으로 섬세하게 설계. **하단 탭에 "기록" 신설**하고, 아래 3개 화면으로 구성한다.
> ① 월 캘린더 + 세션 리스트 → ② 세션 상세 → ③ ··· 옵션 메뉴.

### 현상
`CalendarHeatmap`(최근 16주 잔디, `StatsPage` 내)은 있지만 **날짜를 눌러 그날 기록을 볼 수 없고**, 월 캘린더/세션 상세도 없다. `LogPage`는 오늘 세션만 다룬다.

---

### ① 기록 탭 — 월 캘린더 + 세션 리스트 (레퍼런스 1번 이미지)

**월 캘린더**
- 상단: `< 2026년 7월 >` 월 네비게이션 (이전/다음 달). 가운데 라벨(레퍼런스의 "기록갯수")은 **월별 총 기록 수 배지**로 활용.
- 요일 헤더(일~토), 일요일 시작. 일=빨강, 토=파랑, 이전/다음 달 날짜는 흐리게.
- **날짜 밑 점 마커**로 기록 종류 구분: 🔴 운동 기록(1차 필수) · 🟢 사진 기록 · 🔵 신체 기록. **1차 범위는 운동 점만**, 사진/신체는 향후 확장 자리만 잡아둠(범례는 표시하되 비활성).
- 오늘 날짜 강조, 미래 날짜 비활성.

**하단 세션 리스트** (캘린더 아래 스크롤)
- 카드: `👍 7월 4일, 오전 운동` / `2026-07-04 11:16~12:03, 47분` / 우측 `···`.
- **시간 범위·소요(47분)는 2번 항목의 `started_at/ended_at`이 있어야 표기 가능** → 2번과 함께 진행. 없으면 시간범위 생략하고 날짜만.
- **"오전/오후/저녁 운동" 자동 제목**: `started_at` 시각으로 오전(~12시)/오후(~18시)/저녁 분기. 사용자 지정 제목은 이후.
- 좌측 아이콘(👍 등)은 세션 대표 이모지(기본값 고정, 향후 커스텀).
- 카드 탭 → ② 세션 상세. `···` 탭 → ③ 옵션 메뉴.
- 리스트는 최신순, 월 필터 또는 전체 무한스크롤 중 택1(1차: 선택 월의 세션만).

### ② 세션 상세 (레퍼런스 2번 이미지)

레이아웃 위→아래:
1. **헤더**: `< [제목: 7월 4일, 오전 운동]  ···` + 부제 날짜(`2026.07.04`). 우측 `···` → ③ 옵션.
2. **바디 히트맵**: 앞/뒤 실루엣에 그 세션에서 사용한 부위 강조. **기존 `BodyHeatmap` 컴포넌트 재사용**(현재 통계 기간용 → 단일 세션 입력으로 호출). 세션 내 부위별 볼륨을 히트 강도로.
3. **통계 그리드 (4×2)** — 아래 표 참고. 계산 가능/추정/신규 구분:

   | 지표 | 표기 | 산출 | 상태 |
   |------|------|------|------|
   | WORKOUT | `104th` | 이 프로필의 누적 세션 순번(N번째 운동) | 계산 가능 |
   | CALORIES | `139 KCAL` | **MET × 체중(kg) × 시간(h)** (아래 식) | 프로필 체중 + 2번(시간) 의존 |
   | DURATION | `47분` | `ended_at - started_at` | **2번 의존** |
   | VOLUME | `3234 kg` | Σ(무게×횟수) — `fmtVolume` 활용 | 계산 가능 |
   | EXERCISES | `5` | 운동(entry) 수 | 계산 가능 |
   | SETS | `13세트` | 총 세트 수 | 계산 가능 |
   | REPS | `189회` | 총 반복 수 | 계산 가능 |
   | INTENSITY | `68 kg/분` | VOLUME / DURATION | **2번 의존** |

   > CALORIES·INTENSITY·DURATION은 2번(시간)이 전제. **시간/체중이 없으면 해당 칸 숨기거나 "—" 처리** — 있는 지표부터 출시하고 점진 확장 권장.

   **칼로리 추정 = MET 기반 (프로필 체중 사용)** — *결정: 체중 입력 방식(a)*
   - 식: `kcal = MET × 체중(kg) × 소요시간(시간)`. 소요시간은 `ended_at - started_at`(2번).
     - 예: 체중 60kg, 47분, MET 3.0 → `3.0 × 60 × (47/60) ≈ 141 kcal` (레퍼런스 139와 근접).
   - **MET 값**: 근력운동 근사. 기본 MET ≈ **3.0**(일반적 웨이트). 향후 **INTENSITY(kg/분)로 MET 보정**(가벼움 3.0 / 보통 5.0 / 격렬 6.0 구간화) 가능 — 1차는 상수 3.0로 단순화.
   - **체중 확보**: `profiles.weight_kg`(nullable) 추가.
     - 온보딩에 **선택 입력**(건너뛰기 허용) + **설정 화면(5-G)에서 언제든 입력/수정**.
     - 체중이 없으면 CALORIES 칸을 숨김(다른 지표는 정상 표시).
   - **정확도 주석**: 어디까지나 근사치임을 작은 글씨로 표기(예: "체중 기반 추정").
   - (선택·향후) 체중은 시간에 따라 변함 → 레퍼런스의 "신체 기록"처럼 체중 로그를 두면 세션 시점 체중으로 더 정확. 1차는 현재 프로필 체중 사용.

4. **운동별 카드** (운동 수만큼):
   - 운동명 + `최고 무게: 10kg | 1RM: 15kg`. **1RM은 Epley 추정**(`weight×(1+reps/30)`), 최고무게는 세트 최댓값.
   - 세트를 **원형 배지 행**으로: 원 안=무게, 아래=반복(`15X`=15회). 레퍼런스처럼 가로 나열, 넘치면 줄바꿈.
   - 우측 운동 일러스트(있으면). 없으면 생략.
5. 우하단 **플로팅 "공유하기"** 버튼 → 그룹 피드 공유(기존 `setSessionShared` 재사용). 옵션 메뉴의 "전송"과 역할 정리 필요(공유하기=피드 opt-in, 전송=외부 공유/이미지 등 — 아래 참고).

### ③ ··· 옵션 메뉴 (레퍼런스 3번 이미지, 바텀시트)

- **▶ 이 기록으로 운동 시작** — *(사용자 핵심 요구)* 이 세션의 운동 구성을 복사해 **오늘 새 세션 생성** 후 `LogPage`로 이동. 세트는 비우고 지난기록 autofill로 채우거나(권장), 동일 세트 복제 중 택1. 기존 "루틴으로 시작" 흐름과 유사 → `createSession` + 각 운동 `addEntry` 재사용.
- **✏️ 운동기록 수정** — 과거 세션 편집(세트/무게/메모 수정). `LogPage`를 `date`/`sessionId` 파라미터로 일반화. *(1차엔 읽기전용 후 확장 가능)*
- **🗑 운동기록 삭제** — 세션 삭제(확인 다이얼로그). 연쇄로 entries/sets 삭제.
- **➤ 운동기록 전송** — 외부 공유(요약 이미지/텍스트, Web Share API). 피드 공유(공유하기)와 분리.

## 4. 휴식 타이머 종료 시 알림음

### 현상
`RestTimer`는 0초 도달 시 ✅ "휴식 완료" 텍스트만 바꾼다(`src/components/RestTimer.tsx`). 소리/진동 없음 → 화면을 안 보면 종료를 놓친다.

### 원인 진단
- 종료 감지 로직(`done = remainingMs === 0`)은 있으나 사운드/진동 트리거가 없다.
- **iOS Safari 제약**: 오디오는 사용자 제스처 안에서 최초 재생해야 언락됨. 휴식 타이머는 "세트 완료 탭"(제스처)에서 시작되므로 **그 시점에 AudioContext를 언락**해두면 종료 시 소리 재생 가능.

### 개선안
1. **Web Audio 비프음** — `<audio>`보다 iOS 백그라운드/무음 대응이 나은 Web Audio API로 짧은 비프(또는 짧은 mp3). 세트 완료 시 `AudioContext`를 미리 `resume()`으로 언락.
2. **진동 병행** — `navigator.vibrate([200,100,200])` (Android/Chrome 지원, iOS Safari 미지원이므로 소리와 병행).
3. **트리거 지점** — `RestTimer`에서 `done`이 false→true로 바뀌는 순간 1회 재생 (중복 방지 플래그).
4. **설정** — 음소거 토글 + 볼륨 배려(짧고 부드러운 톤). 무음 스위치/시스템 볼륨 존중.
5. **정확도 보강** — 현재 250ms 인터벌로 카운트다운하나 백그라운드에서 타이머가 throttle되면 알림 지연 가능. 종료 예정 시각(`endsAt`)에 맞춘 별도 알림 스케줄(또는 앱 복귀 시 "이미 지났으면 즉시 울림") 처리.
6. **(선택) 백그라운드 알림** — 화면을 끄거나 다른 앱으로 갔을 때는 소리가 안 울릴 수 있음 → PWA 설치 후 Notification/진동으로 보완 (2차).

### 작업 항목
- [ ] `AudioContext` 언락 유틸 (세트 완료 시 호출)
- [ ] `RestTimer` 종료 전이(false→true) 시 비프 + 진동 1회
- [ ] 알림음 on/off 설정 저장 (localStorage)
- [ ] 백그라운드 throttle 대비 복귀 시 즉시 알림 처리

## 4b. 운동별 휴식시간 저장 (사용자 요구)

### 현상
휴식은 전역 60초 고정(`LogPage.REST_SECONDS`). 타이머에서 `−10초/+10초`로 조정해도(`onAdjust`) **현재 진행 중인 타이머에만 반영되고, 다음 세트/다음 세션엔 다시 60초로 돌아간다.**

### 요구
> 특정 운동에서 휴식을 조정하면(예: −10초) **그 운동의 기본 휴식값으로 저장**되어, 다음부터 그 운동은 무조건 조정된 값으로 시작.

즉 휴식시간은 **운동(exercise) 단위 설정값**이 되어야 하고, 조정이 곧 그 운동의 기본값 갱신이다.

### 개선안
1. **운동별 휴식 저장소** — 프로필×운동 단위로 `rest_seconds`를 영구 저장.
   - **권장(멀티기기 동기화)**: 테이블 `exercise_rest_prefs(profile_id, exercise_id, rest_seconds, updated_at)` + RLS(본인 행만). 기기 변경/복구가 핵심 기능이라 서버 저장이 일관적.
   - **MVP(빠른 적용)**: `localStorage['wt_rest:<exercise_id>']`. 동기화는 안 되지만 즉시 구현. → 이후 테이블로 승격.
2. **타이머 시작 시 값 결정** — 세트 완료로 휴식 시작할 때 `restFor(exerciseId) = pref ?? 전역기본(60)`. `LogPage.toggleComplete`에서 어떤 운동의 세트인지 알 수 있으므로 그 값으로 `restEndsAt` 세팅.
3. **조정 = 기본값 갱신** — 활성 타이머에서 `−10초` 하면 (a)현재 남은 시간에 반영 + (b)**그 운동의 `rest_seconds`를 새 총 휴식값으로 저장**. "다음부터 무조건 적용"을 만족하려면 조정의 의미를 '남은 시간 가감'이 아니라 '이 운동 기본 휴식 = N초'로 확정 저장.
   - 세부: 현재 `onAdjust`는 `restEndsAt`만 민다. 여기에 "활성 타이머의 대상 운동"을 함께 들고 있다가, 조정 시 `기본휴식 += delta`로 계산해 저장하도록 확장. (음수/과대 방지: 예 10~600초 clamp)
4. **UI 힌트** — 타이머에 "이 운동 기본 휴식: 50초" 표기 → 조정이 저장된다는 걸 인지시킴. (선택) 운동 카드/설정에서 운동별 휴식 직접 편집.

### 작업 항목
- [ ] 저장소: `exercise_rest_prefs` 테이블(+RLS) 또는 localStorage MVP
- [ ] `getRestPref/setRestPref(exerciseId)` 유틸 (`workouts.ts` 또는 신규 `restPrefs.ts`)
- [ ] `LogPage`: 활성 휴식의 대상 운동 추적 → 시작 시 운동별 값 사용, `onAdjust` 시 그 운동 값 저장(clamp)
- [ ] `RestTimer`: "이 운동 기본 휴식: N초" 표기
- [ ] (선택) 운동 카드/설정에서 운동별 휴식 직접 설정

## 5. 추가 보완 제안

사용자가 보고하진 않았지만 "앱처럼 매일 쓰는" 경험을 위해 함께 가면 좋은 항목들.

### A. PWA 완성 (1·4번의 전제, 우선순위 높음)
- manifest + 아이콘 + 서비스워커(앱셸 캐싱), iOS safe-area/상태바 점검 → Phase 6와 통합. **1번 저장소 문제와 4번 백그라운드 알림의 근본 해결책**이라 먼저 하는 게 효율적.

### B. 운동 중 화면 꺼짐 방지 (Screen Wake Lock)
- 운동 시작~완료 동안 `navigator.wakeLock`으로 화면 유지 → 세트마다 화면 켜는 불편 제거. 2번 "운동 완료"에서 해제.

### C. 휴식 시간 커스터마이즈
- → **4b로 승격**(운동별 휴식 저장). 추가로 전역 기본 휴식값 설정(운동별 pref 없을 때의 fallback)은 설정 화면(5-G)에 둔다.

### D. 새로고침 시 휴식 타이머 유실 방지
- `restEndsAt`이 상태로만 있어 새로고침하면 사라짐. `endsAt`을 localStorage에 저장 → 복귀 시 남은 시간 복원.

### E. 데이터 내보내기 (JSON/CSV)
- Phase 6 항목. 무료 티어 일시정지/기기 변경 대비 백업 심리적 안전장치.

### F. 연속 기록(스트릭) / 리마인더
- "N일 연속 운동" 표시로 지속 동기. PWA 설치 후 로컬 알림으로 "오늘 운동 안 했어요" 리마인드(2차).

### G. 알림음/진동/휴식 설정을 모을 "설정" 화면
- 1번(복구코드 재확인), 4번(알림음), C(휴식), F(리마인더)가 모두 설정 진입점을 필요로 함 → 통합 설정 페이지 신설 권장.
