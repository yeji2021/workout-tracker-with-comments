# 릴스 투 루틴 (Reel-to-Routine) — 설계 문서

> 2026-07-07 설계 확정. 인스타그램 릴스 / 유튜브 링크로 공유된 운동 루틴 영상을
> Claude가 분석해서 앱의 루틴으로 변환해주는 기능.
> 구현 시점: Phase 6(PWA & 배포) 이후 2차 기능.

## 1. 개요

- 사용자가 운동 루틴 영상 링크(인스타 릴스, 유튜브)를 앱에 등록하면 **큐에 쌓인다.**
- 처리는 **관리자(나) 전용**: 맥의 Claude Code(Cowork) 세션에서 `/릴스처리` 명령으로 큐를 일괄 처리.
- Claude 세션이 로그인된 내 Chrome으로 링크를 직접 열어 **캡션을 먼저 읽고, 정보가 부족하면 영상을 캡처**해서 루틴을 구조화 JSON으로 추출, Supabase에 저장.
- 사용자는 앱에서 추출 결과를 검토·수정한 뒤 "내 루틴으로 저장".

### 왜 이 구조인가

- **인스타그램은 공식 API/스크래핑 경로가 없다.** 데이터센터 IP는 차단되고 yt-dlp류는 유지보수 지옥. 대신 관리자의 로그인된 브라우저로 화면을 보는 방식은 차단·약관 문제가 없다 (개인용 전제).
- **Claude가 직접 화면을 보므로** Anthropic API 키 관리, Edge Function AI 호출, 프레임 dedup 전처리가 전부 불필요. 캡처하면서 "같은 운동이면 스킵" 판단을 세션이 직접 한다.
- **큐 + 수동 트리거**라 백그라운드 워커/크론 인프라가 필요 없고, 비용이 관리자 행동에 비례해 예측 가능.

### 트레이드오프 (인지하고 수용)

- 처리는 맥에서 세션을 실행해야만 일어난다. 앱의 "실행" 버튼은 없고, 상태 표시만 한다.
- 사용자 입장에선 등록 → 처리 완료까지 지연이 있다 (관리자가 돌릴 때까지).
- 관리자 브라우저로 여는 것이므로 비공개 계정 게시물은 관리자가 팔로우 중이어야 보인다.

## 2. 사용자 흐름

### 일반 사용자 (앱)
1. 릴스/유튜브에서 링크 복사 → 앱의 "루틴 가져오기" 화면에 붙여넣기 (+ 선택: 메모)
2. 등록된 링크는 `대기중` 상태로 목록에 표시
3. 처리가 끝나면 `완료` 상태 + 추출된 루틴 미리보기
4. 미리보기에서 운동/세트/횟수 수정 → "내 루틴으로 저장" → 기존 `routines`/`routine_entries`로 들어감
5. 실패 시 `실패` 상태 + 사유 표시 (예: "비공개 계정", "루틴 정보 없음")

### 관리자 (맥, Cowork 세션)
1. `/릴스처리` 실행
2. 세션이 큐를 읽고 링크를 하나씩 처리 (아래 파이프라인)
3. 처리 결과 요약 보고 (성공 N건 / 실패 N건 + 사유)

## 3. 아키텍처

```
[사용자 폰/앱]
  링크 등록 ──▶ routine_imports (status='queued')   ← RLS: 본인 행만 insert/select
  결과 검토 ◀── result(jsonb) 읽기 → routines 테이블로 저장

[관리자 맥, Claude Code 세션]  /릴스처리
  1. Supabase에서 queued 행 조회 (service_role 키, 로컬 보관)
  2. Chrome(claude-in-chrome)으로 링크 열기 — 로그인된 내 브라우저
  3. 캡션/설명란 텍스트 읽기 (get_page_text)
  4. [폴백] 캡션에 루틴 정보 부족 → 영상 재생하며 1~2초 간격 스크린샷
  5. Claude(세션 자신)가 분석 → 루틴 JSON 생성
  6. exercises 시드 테이블과 운동 이름 매칭
  7. result 저장, status='done' (실패 시 'failed' + error)
```

## 4. DB 스키마

```sql
create table routine_imports (
  id          uuid primary key default gen_random_uuid(),
  profile_id  uuid not null references profiles(id),
  url         text not null,
  platform    text not null check (platform in ('instagram', 'youtube', 'other')),
  user_note   text,                        -- 사용자가 남긴 메모 (선택)
  status      text not null default 'queued'
              check (status in ('queued', 'processing', 'done', 'failed')),
  source_text text,                        -- 처리 시 읽은 캡션/설명란 (디버깅·재처리용)
  used_video  boolean default false,       -- 영상 캡처 폴백까지 갔는지
  result      jsonb,                       -- 추출된 루틴 (아래 5절 스키마)
  error       text,                        -- 실패 사유 (사용자에게 그대로 노출)
  created_at  timestamptz not null default now(),
  processed_at timestamptz
);
```

### RLS 정책
- **사용자**: 본인 `profile_id` 행만 insert / select. update·delete 불가 (등록 취소는 queued 상태일 때만 delete 허용 고려).
- **처리 주체**: 관리자 세션은 `service_role` 키로 접근 (RLS 우회). 키는 맥 로컬에만 보관, 절대 repo/프론트에 넣지 않음.
- 플랫폼 판별(`platform`)은 등록 시 URL 패턴으로 클라이언트가 채움.

## 5. 처리 파이프라인 (`/릴스처리` 스킬)

프로젝트 스킬(슬래시 명령)로 작성. 단계:

### 5-1. 큐 조회
- `status='queued'` 행을 오래된 순으로 조회. 처리 시작 시 `processing`으로 마킹.

### 5-2. 캡션 우선 (1차)
- Chrome으로 링크 열기.
- **인스타**: 게시물 캡션 텍스트 읽기. "더 보기" 클릭 필요할 수 있음.
- **유튜브**: 제목 + 설명란(더보기 펼침) + 가능하면 자막.
- 캡션만으로 운동 목록·세트·횟수가 파악되면 **영상 캡처 없이 종료** → 5-4로.

### 5-3. 영상 캡처 폴백 (2차)
- 캡션에 루틴 정보가 없거나 불완전할 때만.
- 영상 재생 → **1~2초 간격 스크린샷**. 릴스는 자동 루프라 놓친 구간은 다음 루프에서 보완.
- 프레임 단위 dedup 전처리는 하지 않는다 — 캡처 주체가 Claude이므로 "아직 같은 운동" 판단을 즉석에서 하고 다음 컷을 기다리면 됨. 릴스 컷 편집은 보통 클립당 2~5초라 이 캡처 해상도로 충분.
- 화면 오버레이 자막("3세트 12회" 등) + 동작 자체 식별을 병행.
- 영상까지 봐도 루틴이 안 나오면 `failed` + error="영상에서 루틴 정보를 찾지 못함".

### 5-4. 구조화 & 운동 매칭
- 결과 JSON 스키마:

```json
{
  "routine_name": "상체 푸시 루틴",
  "source_summary": "영상 요약 한 줄",
  "exercises": [
    {
      "name": "인클라인 덤벨 프레스",
      "matched_exercise_id": "uuid | null",
      "target": "가슴",
      "sets": 4,
      "reps": "8-12",
      "notes": "천천히 내리고 정점에서 1초 정지"
    }
  ]
}
```

- `exercises` 시드 테이블(부위별 50~80개) 목록을 참조해 이름 정규화 + `matched_exercise_id` 매칭. 매칭 실패 시 null → 앱에서 "커스텀 운동으로 추가" 제안.
- `result` 저장, `status='done'`, `processed_at` 기록.

### 5-5. 보고
- 세션 종료 시 처리 요약: 성공/실패 건수, 실패 사유, 영상 폴백까지 간 비율.

## 6. 앱 UI 작업

- [ ] "루틴 가져오기" 화면: 링크 붙여넣기 + 메모 + 등록 (URL 패턴 검증, platform 판별)
- [ ] 내 가져오기 목록: 상태 뱃지 (대기중/처리중/완료/실패)
- [ ] 완료 항목 탭 → 추출 루틴 미리보기 + 편집 (운동 추가/삭제, 세트·횟수 수정, 미매칭 운동은 커스텀 생성 플로우)
- [ ] "내 루틴으로 저장" → `routines`/`routine_entries` insert
- [ ] 실패 항목: 사유 표시 + 삭제

## 7. 구현 순서

1. **DB**: `routine_imports` 테이블 + RLS 마이그레이션 (scripts/에 SQL 추가)
2. **앱**: 링크 등록 + 목록 화면 (처리 없이도 큐가 쌓이는 것 먼저 확인)
3. **스킬**: `/릴스처리` 프로젝트 스킬 작성 — 유튜브부터 (캡션/설명란이 안정적이라 파이프라인 검증에 적합) → 인스타 확장
4. **앱**: 결과 검토 → 루틴 저장 플로우
5. 운영하면서 프롬프트/매칭 정확도 튜닝

## 8. 리스크 & 메모

- **인스타 UI 변경**: 스크래핑이 아니라 화면을 보는 방식이라 파서가 깨지는 문제는 없지만, 캡션 위치 등 페이지 구조가 바뀌면 스킬 지침을 손봐야 할 수 있음. 스킬을 "셀렉터 고정"이 아니라 "화면에서 캡션을 찾아 읽어라" 수준으로 유연하게 쓰면 영향 최소화.
- **service_role 키 보안**: 맥 로컬 환경변수/파일로만 보관. 과거 Supabase 키 노출 사고가 있었으므로(키 로테이션 권장 상태) 특히 주의.
- **비공개 계정**: 관리자가 팔로우하지 않으면 열람 불가 → failed 처리하고 사유 명시.
- **개인용 전제**: 이 파이프라인은 관리자 본인의 브라우저로 본인이 보는 콘텐츠를 정리하는 개인 용도. 서비스 규모로 자동화·확장하지 않는다.
- **처리 지연 안내**: 등록 화면에 "관리자가 주기적으로 처리해요" 문구로 기대치 설정.
